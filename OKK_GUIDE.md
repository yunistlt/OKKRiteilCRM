# Руководство по системе ОКК

Мы договорились о следующей терминологии:
1.  **Правила**: Механизм на странице «Настройка правил». Это динамические проверки, которые можно создавать и менять в базе данных.
2.  **Чеклист контроля**: Колонки на главном дашборде (Семён, Игорь, Максим). Это жестко прописанная логика в коде (`okk-evaluator.ts`).

---

## Как не путаться: Чеклист vs Правила

1.  **Чеклист контроля (Общая сводка)**:
    *   Это колонки на дашборде (ТЗ получено, ИНН есть и т.д.).
    *   Они нужны для **мониторинга** и формирования общей оценки (%) за ведение заказа.
    *   Если в чеклисте стоит «Красный крестик», это портит рейтинг, но **само по себе** не создает отдельное «Нарушение» в списке алармов.

2.  **Правила (Ваш механизм управления)**:
    *   Это то, что вы настраиваете на странице «Настройка правил».
    *   Они создают **Нарушения** (Alerts/Violations).
    *   Если вы хотите, чтобы при отсутствии ТЗ менеджеру выписывался «штраф» или падало уведомление, вы создаете это как **Правило**.

---

## Создание Правил (динамических)
Правила хранятся в таблице `okk_rules` в Supabase. Каждое правило состоит из трех частей:
1.  **Триггер (Trigger)**: Когда запускать проверку (например, при смене статуса).
2.  **Условия (Conditions)**: Что именно проверять (наличие полей, анализ текста через AI и т.д.).
3.  **Параметры**: Штрафные баллы, критичность, уведомления в Telegram.

---

## Как создать правило через SQL

Вы можете добавить новое правило напрямую в базу данных:

```sql
INSERT INTO okk_rules (code, name, description, entity_type, logic, points, severity, is_active, notify_telegram)
VALUES (
  'rule_my_custom_check', 
  'Моя проверка', 
  'Описание того, что мы проверяем', 
  'order', -- тип сущности (order, call или event)
  '{
    "trigger": {
      "block": "status_change",
      "params": { "target_status": "zapros-kontaktov" }
    },
    "conditions": [
      {
        "block": "field_empty",
        "params": { "field_path": "INN" }
      }
    ]
  }'::jsonb,
  10, -- Штрафные баллы
  'medium', -- Критичность: low, medium, high, critical
  true, -- Активно ли правило
  true  -- Отправлять ли в Telegram
);
```

---

## Доступные блоки (Logic Blocks)

### Триггеры (Trigger blocks):
- `status_change`: Срабатывает при смене статуса. Параметр `target_status`.
- `field_change`: Срабатывает при изменении конкретного поля.
- `new_call_transcribed`: Срабатывает, когда появляется новая расшифровка звонка.

### Условия (Condition blocks):
- `field_empty`: Проверяет, пустое ли поле. Параметр `field_path`.
- `semantic_check`: **AI-проверка**. Отправляет текст (комментарий или звонок) в GPT. Параметр `prompt` (инструкция для робота).
- `time_elapsed`: Проверяет время нахождения в статусе.

---

## Советы по созданию
1.  **Семантические проверки**: Используйте `semantic_check` для сложных условий, которые нельзя проверить просто «на пустоту» (например, «проверь, вежлив ли менеджер»).
2.  **Коды статусов**: Используйте системные коды RetailCRM (например, `zapros-kontaktov`, а не «Запрос контактов»).
3.  **Тестирование**: После создания правила нажмите кнопку **RUN** в интерфейсе ОКК для конкретного заказа, чтобы проверить, как оно работает.
