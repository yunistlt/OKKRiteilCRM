# План готовности раздела «Корпоративный мессенджер»

## Текущая оценка готовности

- Оценка: около 92% до внутреннего production-ready состояния.
- Статус: основная продуктовая, серверная и push-функциональность уже реализована; desktop web push подтверждён в production, telegram-like mobile shell и avatar support для пользователей/групп уже внедрены, включая signed delivery из private storage, незакрытый контур сводится к production smoke-check на реальных сценариях и mobile-устройствах.
- Что уже есть: страница раздела, список чатов, просмотр сообщений, отправка сообщений, создание чатов, управление участниками группы, бейдж непрочитанных, Supabase Realtime-подписки, базовая схема БД для чатов/сообщений/участников, user avatars и group avatars с private-storage выдачей через signed URL, корректным signed upload flow и клиентским уменьшением тяжёлых avatar images перед отправкой.

### 6. Push-уведомления для desktop и mobile

#### Рекомендуемая реализация

- Базовый путь для текущего продукта: PWA + Web Push для desktop и mobile web.
  Это самый быстрый путь для текущего Next.js-приложения без запуска отдельной мобильной разработки с нуля.

- Рекомендуемый fallback: внутренний unread badge, app badge, title badge и in-app polling без внешних мессенджеров.
  Это закрывает случаи, когда браузерный push недоступен, отключён пользователем или работает нестабильно на части устройств.

- Отдельное mobile-приложение стоит планировать только как второй этап.
  Оно оправдано, если появятся требования к более надёжным background notifications, richer mobile UX, offline-first и глубоким platform-specific сценариям.

#### Рекомендуемая архитектура v1

- [x] Реализовать Web Push subscription flow в браузере: запрос разрешения, получение subscription, сохранение на сервере.

  Email можно оставить только как редкий резервный канал для важных событий, но не как основной механизм новых сообщений.

#### Отдельный mobile app: не входит в текущий v1 scope

- Для текущего релиза базовый путь зафиксирован как PWA + Web Push.
- Переход к отдельному mobile app рассматривается только после production-подтверждения, что web-only схема не закрывает iPhone/background/UX-требования.
- Если отдельный mobile app когда-либо стартует, push provider слой нужно выделять отдельно через APNs/FCM и не смешивать с web-only реализацией.
- Notification domain layer должен остаться единым для web push и будущего native push.
