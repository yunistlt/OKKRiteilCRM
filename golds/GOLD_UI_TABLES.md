# GOLD: Стандарт Интерфейсных Таблиц (UI)

> [!IMPORTANT]
> **SCOPE:** Этот документ описывает исключительно правила визуального отображения, верстки и UX таблиц для пользователя. 
> **НЕ ИСПОЛЬЗОВАТЬ** для проектирования схем базы данных или SQL-запросов.

Этот документ определяет "Золотые правила" реализации таблиц в интерфейсе проекта для обеспечения максимальной читаемости и функциональности.


## 1. Визуальные правила (UI)
*   **Sticky Header:** Заголовок таблицы всегда зафиксирован сверху при скролле.
*   **Zebra Striping:** Чередование цвета строк для облегчения горизонтального чтения (используем очень бледный серый).
*   **Hover State:** Строка должна подсвечиваться при наведении, чтобы пользователь не терял фокус.
*   **Cell Padding:** Минимум 12px по вертикали и 16px по горизонтали.
*   **Status Indicators:** Использование иконок (Lucide) вместо текста для бинарных состояний (V/X) с цветовой кодировкой (Green/Red).

## 2. Функциональные правила (UX)
*   **Horizontal Scroll:** Если колонок слишком много, таблица должна скроллиться горизонтально, но первая колонка (Заказ/ID) должна оставаться зафиксированной слева.
*   **Grouping:** Категории критериев должны иметь визуальное объединение в заголовке (Colspan).
*   **Empty States:** Если данных в ячейке нет, ставится прочерк `—` серого цвета, а не пустое место.
*   **Alignment:** 
    *   Текст — по левому краю.
    *   Числа/Суммы — по правому краю (или по центру, если это баллы).
    *   Иконки статусов — строго по центру.

## 3. Технические правила (Code)
*   **Dynamic Columns:** Колонки не должны быть жестко зашиты в коде. Они приходят из конфига (JSON).
*   **Memoization:** Используем `React.memo` для строк, если таблица превышает 50 строк, чтобы избежать лагов при рендеринге.
*   **Data Formatting:** Суммы всегда форматируются с разделением разрядов (100 000 руб.) и сокращением до `к` (100к), если места мало.

## 4. Цветовая шкала Оценок (Scoring)
*   **90-100%:** Эспрессо-зеленый (High Success).
*   **70-89%:** Светло-зеленый (Pass).
*   **50-69%:** Желтый/Оранжевый (Warning).
*   **< 50%:** Красный (Fail).

## 5. Фильтрация и Поиск (Filtering)
*   **Локальные фильтры (Колонки):** В заголовке каждой колонки может располагаться иконка воронки для быстрой фильтрации по значениям.
*   **Режимы фильтрации:** Логика фильтров должна поддерживать два режима:
    *   *Клиентский (Client-side):* Поиск только по уже загруженным на страницу строкам.
    *   *Серверный (Server-side):* Глобальный поиск по всем данным таблицы в БД.
*   **Блок расширенных фильтров:** Над таблицей располагается панель для глобальных/частых фильтров.
*   **Кастомизация (Шестеренка):** Состав блока расширенных фильтров настраивается. По клику на иконку "шестеренки" (настройки таблицы) пользователь отмечает чекбоксами те колонки, фильтры по которым он хочет вынести в этот верхний блок.

## 6. Продвинутый UX (по опыту B2B/CRM)
*   **Пресеты (Сегменты) фильтров:** Возможность сохранить выбранный набор расширенных фильтров как "Вкладку" над таблицей (например: "Новые нарушения", "Мои проверки").
*   **Индикация активности:** Если фильтр применен, иконка воронки в колонке обязана менять цвет (акцентироваться), а также должна быть доступна кнопка "Сбросить все".
*   **Настройка видимости колонок:** В меню "шестеренки" помимо фильтров должна быть возможность скрыть/показать колонки и изменить их порядок.
*   **Массовые действия (Bulk Actions):** Добавление чекбоксов в первую колонку. При выборе строк над таблицей (или внизу экрана) появляется плавающая панель с доступными массовыми операциями.
*   **Быстрый просмотр (Quick View):** Клик по строке (не по конкретной кнопке или ссылке внутри нее) должен открывать боковую панель (Сайдбар) с деталями сущности, чтобы не ломать контекст списка и не уводить пользователя на другую страницу.
*   **Пагинация:** Использование понятной постраничной навигации с селектором количества строк на страницу (20, 50, 100). Для сложных интерфейсов это удобнее, чем бесконечный скролл (Infinite scroll), так как сохраняет предсказуемость высоты страницы.
*   **Масштаб таблицы (Zoom):** Для широких таблиц с большим числом колонок пользователь должен иметь возможность менять масштаб (как Ctrl +/− в Google Таблицах), чтобы уместить на экране больше данных.
    *   *Контрол:* Компактный блок `−  NN%  +` рядом с кнопкой «Колонки» над таблицей. Клик по проценту сбрасывает масштаб в 100%.
    *   *Хоткеи:* `Ctrl/Cmd + =` / `Ctrl/Cmd + −` меняют масштаб с шагом, `Ctrl/Cmd + 0` сбрасывает в 100% (как в Google Таблицах). Системный зум браузера при этом перехватывается (`preventDefault`).
    *   *Диапазон:* от 50% до 130% с шагом 10%; кнопки `−/+` блокируются на границах.
    *   *Реализация:* CSS-свойство `zoom` применяется к самому `<table>` (а не к скролл-контейнеру) — так горизонтальный скролл и sticky-заголовок продолжают работать корректно. `transform: scale()` НЕ использовать (ломает sticky и расчёт ширины скролла).
    *   *Персистентность:* выбранный масштаб сохраняется в `localStorage` (ключ `okk_table_zoom`) и восстанавливается при перезагрузке.
