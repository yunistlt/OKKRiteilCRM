# GOLD: Архитектура Фронтенда и Управление Состоянием

Этот документ описывает подходы к написанию клиентского кода (React / Next.js) для поддержания чистоты, скорости и масштабируемости интерфейса.

## 1. Разделение Компонентов
*   **Server Components по умолчанию:** В Next.js App Router все компоненты по умолчанию серверные. Делайте компонент клиентским (`"use client"`) только если ему нужны хуки (`useState`, `useEffect`), браузерные API или обработчики событий (`onClick`).
*   **Глупые и Умные компоненты:** 
    *   *UI-компоненты (Dumb):* Кнопки, карточки, инпуты. Не знают ничего о бизнес-логике и базе данных. Получают данные только через `props`.
    *   *Контейнеры (Smart):* Страницы и крупные блоки. Выполняют запросы к API/БД и передают данные вниз в UI-компоненты.

## 2. Управление состоянием (State Management)
*   **Локальное состояние:** Используйте `useState` для UI-состояний (открыто ли модальное окно, значение поля ввода).
*   **Глобальное UI состояние:** Если состояние нужно в нескольких местах (например, открыт ли сайдбар, данные текущего юзера), используйте `Zustand` или `React Context`. Избегайте Prop Drilling (передача пропсов глубже 2-3 уровней).
*   **Серверное состояние:** Данные из API (список заказов, отчет) — это серверное состояние. Запрещено копировать их в глобальный стейт типа Redux. Используйте `SWR` или `React Query` на клиенте, или прямые запросы в Server Components.

## 3. Обработка Загрузки и Ошибок (UX)
*   **Skeleton Loaders:** При загрузке данных показывайте скелетоны, а не пустой белый экран или прыгающий лоадер, чтобы сохранить структуру (Grid) согласно стандартам Metro UI (GOLD_DESIGN_UX).
*   **Graceful degradation:** Если компонент упал с ошибкой, это не должно ломать всю страницу. Используйте `error.js` (Error Boundaries) в Next.js.

## 4. Оптимизация производительности
*   **Импорты:** Импортируйте иконки (Lucide) и тяжелые библиотеки точечно, чтобы не раздувать размер бандла.
*   **Мемоизация:** Используйте `React.memo`, `useMemo` и `useCallback` только для тяжелых вычислений или в больших таблицах (согласно GOLD_TABLE_STANDARDS). Преждевременная оптимизация вредна.

## 5. Именование (Naming Conventions)
*   **Компоненты:** `PascalCase` (например, `SidebarMenu.jsx`).
*   **Хуки:** Обязательно с префикса `use` в `camelCase` (например, `useAuth.js`).
*   **Обработчики событий:** Начинаются с `handle` (например, `handleSubmit`, `handleRowClick`).
*   **Пропсы событий:** Начинаются с `on` (например, `onClick`, `onSave`).

## 6. Двухуровневый интерфейс (Tiered UI)
*   **Customer Dashboard (`/`):** Основной интерфейс для клиентов. Ограничен данными одного `tenant_id`.
*   **Super Admin Panel (`/admin`):** Глобальный интерфейс для управления SaaS-платформой. Доступен только пользователям с ролью `superadmin`.
*   **Изоляция Layout:** Для `/admin` используется отдельный `layout.js` с системным сайдбаром и специфической цветовой гаммой (Deep Dark / Purple акценты), чтобы визуально отличить управление системой от использования сервиса.
