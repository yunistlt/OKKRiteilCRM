# GOLD: Стандарт Базы Данных и SQL

> [!IMPORTANT]
> **SCOPE:** Этот документ описывает правила проектирования архитектуры БД (PostgreSQL) и написания SQL-запросов. 
> **НЕ ИСПОЛЬЗОВАТЬ** для решения вопросов отображения данных (верстки таблиц). См. [GOLD_UI_TABLES.md](./GOLD_UI_TABLES.md).

Этот документ описывает правила проектирования архитектуры БД (PostgreSQL) и написания SQL-запросов.


## 1. Соглашение об именовании (Naming Conventions)
*   **База данных (Таблицы и Колонки):** Всегда используется `snake_case` (например: `users`, `tenant_id`, `created_at`). Названия таблиц — во множественном числе (`users`, а не `user`).
*   **Код (JavaScript/TypeScript):** Переменные и свойства объектов всегда в `camelCase`.
*   **Маппинг:** При получении данных из БД в API, преобразовывайте `snake_case` в `camelCase` перед отправкой на фронтенд, чтобы код оставался консистентным.

## 2. Безопасность и Изоляция (Multi-tenancy)
*   **Tenant ID:** 99% таблиц должны иметь колонку `tenant_id` (FOREIGN KEY на таблицу `tenants`).
*   **Обязательный фильтр:** Запрещено писать `SELECT * FROM orders`. Всегда должно быть `SELECT * FROM orders WHERE tenant_id = $1`. Исключение — таблицы системных настроек и пользователи с ролью `superadmin`.
*   **Super Admin:** Пользователи с ролью `superadmin` имеют право выполнять глобальные запросы без фильтрации по `tenant_id` для управления всей платформой.

## 3. Удаление данных (Soft Deletes)
*   **Никакого DELETE:** В бизнес-критичных таблицах (пользователи, проверки, отчеты, интеграции) прямое удаление строки (`DELETE FROM...`) строго запрещено.
*   **Флаг удаления:** Используйте колонку `deleted_at TIMESTAMP`. Для удаления устанавливаем текущее время. Для выборки активных записей всегда добавляем `WHERE deleted_at IS NULL`.

## 4. Время и Таймзоны
*   **Только UTC:** Любые даты хранятся в БД исключительно в формате UTC (`TIMESTAMP WITH TIME ZONE`).
*   **Локализация:** Преобразование времени в локальный часовой пояс клиента (например, Москва +3) происходит только на этапе отображения на фронтенде.
*   **Системные поля:** Каждая таблица обязана иметь поля `created_at` (по умолчанию `NOW()`) и `updated_at`.

## 5. Производительность (Индексы)
*   Каждый внешний ключ (Foreign Key) должен быть проиндексирован.
*   Поля, по которым часто идет поиск или фильтрация (`tenant_id`, `status`, `email`), должны иметь индексы (`CREATE INDEX`).

## 6. Запросы (node-postgres / pg)
*   **Параметризация:** Использование шаблонных строк или конкатенации для SQL-запросов недопустимо из-за риска SQL-инъекций. Обязательно использование `$1, $2`.
*   **Освобождение пула:** При использовании `pool.connect()` (для транзакций) обязательно вызывайте `client.release()` в блоке `finally`.
*   **Транзакции:** Все операции, состоящие из 2 и более инсертов/апдейтов, должны быть обернуты в `BEGIN` и `COMMIT`.
