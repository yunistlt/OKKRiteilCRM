# GOLD: Процесс разработки и Git

Этот документ описывает правила работы с репозиторием и кодом для обеспечения командной работы и легкого отслеживания истории изменений.

## 1. Стратегия Ветвления (Branching)
*   `main` (или `master`) — рабочая ветка, зеркало продакшена. Код здесь всегда стабилен.
*   Прямые коммиты в `main` ЗАПРЕЩЕНЫ.
*   Все работы ведутся в отдельных ветках. Формат названий веток:
    *   `feature/название-фичи` (например: `feature/invite-system`)
    *   `bugfix/описание-бага` (например: `bugfix/table-scroll`)
    *   `hotfix/критический-баг-на-проде`

## 2. Правила Коммитов (Conventional Commits)
Используем семантическое именование коммитов. Это позволяет легко читать историю:
*   `feat: добавлены инвайт-ссылки для новых юзеров` — новый функционал.
*   `fix: исправлено сползание таблицы на мобильных` — исправление бага.
*   `refactor: оптимизирован роут регистрации` — изменение кода без добавления функционала и фикса багов.
*   `style: отключены скругления кнопок (Metro UI)` — изменения внешнего вида.
*   `docs: обновлен GOLD_SECURITY_STANDARDS` — работа с документацией.
*   `chore: обновление зависимостей` — рутинные задачи, конфиги.

## 3. Чек-лист перед созданием Pull Request (Merge Request)
1. Код соответствует архитектуре и дизайну (проверено по всем файлам `GOLD_...`).
2. Нет хардкода: токены и ключи вынесены в переменные окружения (`.env`).
3. В консоли браузера и терминале сервера нет предупреждений (Warnings) и ошибок.
4. Оставлены комментарии `// TODO: ...` только в тех местах, где явно требуется технический долг, и создана задача в трекере.
