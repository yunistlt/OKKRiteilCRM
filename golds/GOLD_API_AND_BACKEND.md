# GOLD: Стандарт разработки API и Бэкенда

Этот документ описывает правила проектирования и разработки серверной части на базе Next.js App Router (Route Handlers и Server Actions).

## 1. Формат ответов API (JSON Contract)
*   **Единообразие:** Все API-ответы должны иметь строго определенную структуру. Фронтенд всегда должен знать, что парсить.
*   **Успешный ответ:** 
    ```json
    { "data": { "id": 1, "name": "ОКК" }, "error": null }
    ```
*   **Ответ с ошибкой:** 
    ```json
    { "data": null, "error": { "message": "Неверный пароль", "code": "AUTH_FAILED" } }
    ```

## 2. HTTP Статусы
Запрещено возвращать `200 OK`, если произошла логическая ошибка.
*   `200 OK` — Успех.
*   `201 Created` — Успешное создание ресурса (например, зарегистрирован пользователь).
*   `400 Bad Request` — Ошибка валидации входящих данных (неверный email, пустое поле).
*   `401 Unauthorized` — Пользователь не авторизован (нет сессии или токена).
*   `403 Forbidden` — Авторизован, но нет прав на это действие (RBAC).
*   `404 Not Found` — Ресурс не найден (или не принадлежит этому `tenant_id`).
*   `429 Too Many Requests` — Превышен лимит запросов.
*   `500 Internal Server Error` — Непредвиденная ошибка сервера (падение БД, баг в коде).

## 3. Обработка ошибок (Error Handling)
*   **Try/Catch:** Вся бизнес-логика в API роутах должна быть обернута в блок `try/catch`.
*   **Скрытие деталей:** Запрещено отдавать на фронтенд "сырые" ошибки базы данных или стектрейсы. Пользователь должен видеть понятные сообщения, а полная ошибка пишется в консоль/логгер.

## 4. Next.js App Router Специфика
*   **Route Handlers (`app/api/.../route.js`):** Используются для внешних интеграций (вебхуки от CRM) и для клиентских компонентов (React Client Components), которым нужно получить данные после рендера.
*   **Server Actions:** Для форм и мутаций данных из клиентских компонентов предпочтительно использовать Server Actions, так как они обеспечивают типобезопасность и автоматическую инвалидацию кэша.
*   **Server Components:** Получение данных для отрисовки страницы должно происходить напрямую в серверных компонентах без вызова собственных API (обращаемся к БД напрямую).

## 5. Валидация
*   Все данные, приходящие от клиента (`req.json()` или `FormData`), считаются небезопасными.
*   Перед любой операцией с БД данные обязаны пройти валидацию (в идеале использовать библиотеки вроде `Zod` для декларативной валидации).
