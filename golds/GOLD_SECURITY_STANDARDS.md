# GOLD: Стандарты Безопасности и Авторизации

Этот документ определяет "Золотые правила" обеспечения безопасности данных, контроля доступа и процессов аутентификации в проекте. Любой новый код, затрагивающий данные клиентов или API, обязан соответствовать этим стандартам.

## 1. Регистрация и Доступ (Invite-Only Model)
*   **Закрытая регистрация:** Самостоятельная (публичная) регистрация в сервисе СТРОГО ЗАПРЕЩЕНА. Любые публичные эндпоинты регистрации должны возвращать ошибку.
*   **Система Инвайтов:** Создание нового аккаунта возможно исключительно по уникальной сгенерированной ссылке-приглашению (Invite URL).
*   **Права на генерацию:** Ссылки генерирует либо собственник платформы (Суперадмин) для подключения новой компании, либо уполномоченный сотрудник клиента (Администратор) для добавления коллег в свой воркспейс.
*   **Жизненный цикл инвайта:** Пригласительная ссылка обязана быть одноразовой (флаг `is_used = true` после успешной активации) и иметь ограниченный срок жизни (поле `expires_at`).

## 2. Изоляция Данных (Multi-Tenancy)
*   **Tenant ID:** Архитектура БД подразумевает хранение всех клиентов в общих таблицах. КАЖДЫЙ запрос к базе (SELECT, UPDATE, DELETE), затрагивающий пользовательские данные, ОБЯЗАН содержать фильтрацию по `tenant_id` (ID компании).
*   **Бэкенд-фильтрация:** Запрещено полагаться на `tenant_id`, приходящий с фронтенда. ID компании должен извлекаться строго из защищенной сессии сервера (NextAuth session) для предотвращения IDOR (Insecure Direct Object Reference).

## 3. Аутентификация и Сессии
*   **Хеширование паролей:** Пароли в базе данных хранятся строго в виде хешей (используется `bcryptjs` с минимум 10 "солями"/salt rounds). Работа с паролями в открытом виде запрещена.
*   **Провайдер:** Для управления сессиями используется `NextAuth.js`. Сессии должны быть защищены флагом HTTPOnly для предотвращения перехвата XSS-атаками.
*   **Верификация почты:** При входе (особенно с новых устройств или при первом доступе) используется двухэтапная проверка через 6-значный код на email (`verification_codes`), срок действия которого строго ограничен (например, 15 минут).

## 4. Защита API и Транзакции
*   **Валидация входящих данных:** Все данные из `request.json()` (email, пароли, имена) должны проходить санитаризацию (удаление пробелов `trim()`, приведение email к `toLowerCase()`) и валидацию форматов (RegEx) до начала бизнес-логики.
*   **SQL-инъекции:** Использование сырых строк в запросах категорически запрещено. Все запросы к PostgreSQL должны быть параметризованными (через `$1, $2, $3`).
*   **Транзакционность:** Любые составные операции (например, создание `tenant` + создание `user` + обновление `invites`) обязаны выполняться внутри транзакций (`BEGIN`, `COMMIT`, `ROLLBACK`). Это защищает от появления "повисших" данных в случае ошибки на одном из шагов.

## 5. Ролевая модель (RBAC - Role-Based Access Control)
*   **Уровни доступа:**
    *   `superadmin`: Глобальный доступ к платформе, управление всеми компаниями (tenants) и системными настройками. Доступ к `/admin`.
    *   `admin`: Владелец аккаунта компании. Управление настройками, сотрудниками и биллингом внутри своего `tenant_id`.
    *   `rop`, `controller`, `manager`: Линейные роли внутри компании с ограниченным доступом к данным.
*   **Серверная проверка (Server-side Enforcement):** Скрытие элементов интерфейса на фронтенде не является защитой. Каждый защищенный API-эндпоинт обязан проверять текущую роль пользователя в сессии перед выдачей данных. Для `/admin` проверка роли `superadmin` обязательна на уровне Middleware.

## 6. Безопасность интеграций и ИИ
*   **Ключи сторонних сервисов:** API-ключи (OpenAI, amoCRM, Bitrix24, RetailCRM) хранятся в БД в таблице `tenants`. Доступ к эндпоинту получения этих настроек должен быть строго ограничен ролью `admin` конкретного `tenant_id`.
*   **Защита Промптов (Prompt Injection):** Системные промпты должны конструироваться на сервере. Недопустимо передавать сырые пользовательские данные прямо в системный контекст LLM без экранирования или строгих инструкций игнорировать попытки взлома промпта клиентом.
