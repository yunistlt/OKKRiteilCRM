# Корпоративный мессенджер: модель доступа

Статус: зафиксированная серверная модель доступа для текущей реализации messenger.

## 1. Базовый принцип

- Серверные маршруты messenger не полагаются только на RLS, потому что используют серверный Supabase-клиент.
- Критические проверки доступа выполняются явно через membership в `chat_participants`.
- Вложение считается доступным только если:
  - пользователь состоит в чате,
  - путь вложения принадлежит именно этому чату,
  - storage URL выдаётся через защищённый endpoint, а не через публичную ссылку.

## 2. Direct-chat

- Читать историю direct-чата может только участник direct-чата.
- Отправлять сообщения в direct-чат может только участник direct-чата.
- Скачивать вложения direct-чата может только участник direct-чата.
- Создание дубля direct-чата не допускается: перед созданием выполняется серверный поиск уже существующего диалога между двумя участниками.

## 3. Group-chat

- Читать историю group-чата может только текущий участник group-чата.
- Отправлять сообщения в group-чата может только текущий участник group-чата.
- Добавлять и удалять участников может только admin группы.
- Переименовывать и удалять group-чата может только admin группы.
- Выйти из group-чата пользователь может только как участник этой группы.

## 4. Removed member

- После удаления из группы пользователь теряет доступ к новым чтениям через API сообщений.
- После удаления из группы пользователь теряет доступ к скачиванию вложений этого чата через защищённый attachments endpoint.
- Presence и push-поведение больше не должны учитывать его как валидного получателя по этому чату.

## 5. Non-member

- Пользователь вне чата получает `403 Forbidden` на чтение `/api/messenger/messages?chat_id=...`.
- Пользователь вне чата получает `403 Forbidden` на защищённый `/api/messenger/attachments?...`.
- Пользователь вне чата не может отправлять сообщения в чужой чат.
- Пользователь вне чата не может менять состав участников или состояние чужого group-чата.

## 6. Вложения

- Сервер валидирует mime type и размер файла до выдачи signed upload URL.
- Имя файла нормализуется сервером.
- Path traversal и межчатовый доступ режутся через проверку chat-scoped path.
- При удалении сообщения или чата связанные вложения удаляются из storage.

## 7. Проверка

- Ручная проверка сценариев описана в [OKK_CORPORATE_MESSENGER_SMOKE_CHECK.md](OKK_CORPORATE_MESSENGER_SMOKE_CHECK.md).
- Обязательные smoke-paths:
  - direct-chat participant
  - group-chat participant
  - removed member
  - non-member
  - attachment download access
