# ТЗ: детерминированные роли по стерео-каналам

Дополнение к `STT_WORKER_SPEC.md`. Цель — убрать путаницу ролей (Менеджер/Клиент) во **всех**
звонках. Сейчас роли проставляет LLM по тексту (gpt-4o-mini) — на сложных/коммутаторных звонках
ошибается. Телфин пишет звонок в **2 канала** (оператор и клиент раздельно), значит роли можно
проставлять **детерминированно по каналу**, без угадывания.

## Разделение ответственности

- **Воркер (Timeweb):** определяет, что запись 2-канальная, транскрибирует **каждый канал
  отдельно** и присылает сегменты с номером канала. **Роли воркер НЕ решает** (он не знает, какой
  канал — оператор).
- **ОКК (наша сторона):** определяет, какой канал — оператор (по содержимому: компания/менеджер
  всегда представляется — это надёжное бинарное решение), мержит каналы по таймкодам и проставляет
  `Менеджер:`/`Клиент:` детерминированно по каждой реплике.

Турн-атрибуция (кто что сказал) становится точной — она задаётся каналом, а не догадкой. Единственное
«умное» решение — бинарное «какой из двух каналов оператор» — на порядок надёжнее полной диаризации.

---

## Что делает ВОРКЕР (это и есть ТЗ для разработчика)

### 1. Определить число каналов
После скачивания записи — определить каналы (`ffprobe`/декод). 2 канала → стерео-путь, 1 → моно-путь
(как сейчас).

### 2. Стерео-путь (2 канала)
- Разделить на 2 моно-дорожки (например `ffmpeg -i in.mp3 -filter_complex channelsplit=channel_layout=stereo[L][R] ...`,
  или `pydub`: `AudioSegment.split_to_mono()`).
- Транскрибировать **каждую** дорожку отдельно (faster-whisper, `language="ru"`), получить сегменты
  с таймкодами по каждому каналу.
- Отправить в `POST /api/stt/result`:

```json
{
  "call_id": "....",
  "status": "done",
  "channels": 2,
  "segments": [
    { "start": 0.00, "end": 2.10, "text": "Компания ЗМК, добрый день", "channel": 0 },
    { "start": 2.30, "end": 3.90, "text": "Здравствуйте",            "channel": 1 },
    { "start": 4.10, "end": 7.50, "text": "Вы по заказу звонили?",   "channel": 0 }
  ]
}
```
- `channel` — **0 или 1** (левый/правый). Роли НЕ проставлять. Сегменты можно слать вперемешку —
  ОКК сам отсортирует по `start`.

### 3. Моно-путь (1 канал) — без изменений
Как в `STT_WORKER_SPEC.md`: `{ call_id, status:"done", text, segments? }`. ОКК доразметит роли
текущим способом (LLM).

### 4. Псевдо-стерео (необязательно, но желательно)
Если оба канала несут **одинаковый** звук (моно в стерео-контейнере — встречается), детектировать
это (каналы почти идентичны) и слать как **моно** (`text`), чтобы не плодить дубли реплик. Если
детектить сложно — не критично, ОКК подстрахуется (см. ниже).

## Что делает ОКК (наша сторона — реализуем сами)

`POST /api/stt/result` при `channels: 2` и сегментах с `channel`:
1. Определяем **канал оператора** по содержимому каналов (бинарно, надёжно: оператор представляется
   компанией/именем, ведёт по скрипту). Канал оператора → `Менеджер`, второй → `Клиент`.
2. Мержим сегменты обоих каналов по `start`, склеиваем подряд идущие реплики одной роли.
3. Пишем готовый транскрипт `Менеджер: ... / Клиент: ...` — **детерминированно по каналу**, без
   текстовой диаризации.
4. Подстраховка от псевдо-стерео: если каналы почти идентичны (дубль) — откатываемся на обычную
   моно-обработку.

При `channels: 1` или отсутствии каналов — текущий fallback (LLM-диаризация по тексту).

---

## Итог
- Роли становятся **точными во всех типах звонков** (продажи, входящие, коммутатор) — за счёт
  физического разделения каналов.
- Воркеру достаточно: детект каналов → split → транскрибация каждого → отдать сегменты с `channel`.
- Стерео уже включён на Телфине и идёт на дневных звонках — можно внедрять.
