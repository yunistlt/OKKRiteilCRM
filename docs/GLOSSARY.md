# Глоссарий терминов OKKRiteilCRM

**Дата:** Май 2026  
**Версия:** 1.0  
**Статус:** Живой документ (обновляется по мере добавления новых терминов)

---

## А

### Account (Аккаунт)
- **Определение:** Организация/клиент в multi-tenant системе, использующий платформу
- **Связь:** Каждый аккаунт имеет своих пользователей, интеграции и изолированные данные
- **Пример:** "Клиент ООО Рога и Копыта использует отдельный аккаунт"
- **Поле в БД:** `accounts.id`, `orders.account_id`

### Audit trail (Трейл аудита)
- **Определение:** Полная история всех действий пользователя (create, update, delete)
- **Содержит:** Кто, что, когда, старые значения, новые значения
- **Таблица:** `audit_events`
- **Критичность:** CRITICAL для compliance

### Agent (ИИ-агент)
- **Определение:** Специализированный ИИ компонент с одной ролью
- **Примеры:** Анна (анализ), Максим (аудит), Дарья (legal helpdesk)
- **Принцип:** Каждый агент читает из одной таблицы, пишет в одну (или очередь)

---

## Б

### Backlog (Очередь задач)
- **Определение:** Количество необработанных job'ов в очереди
- **Метрика:** SLA если backlog > N за время T
- **Мониторинг:** `/api/settings/system-status`

### Batch operation (Пакетная операция)
- **Определение:** Обработка N записей в одном запросе (ОНО ЖЕ "批量")
- **Примерно:** Раньше `/api/okk/run-all` пересчитывал все 500 заказов за раз
- **Сейчас:** Deprecated в пользу event-driven single-order refresh
- **Примеры:** Legacy `/api/sync/retailcrm` (limit=50 на запрос)

---

## В

### Webhook (Вебхук)
- **Определение:** HTTP callback от внешней системы (Telphin, RetailCRM) в нашу API
- **Примеры:**
  - Telphin: `POST /api/webhooks/telphin/call-ended`
  - RetailCRM: `POST /api/webhooks/retailcrm/order-changed`
- **Гарантия:** Доставка не гарантирована → fallback poller

---

## Г

### Golden fixture (Золотая фиксация)
- **Определение:** Набор реальных тестовых данных для regression suite
- **Примеры:**
  - 20-30 контрактов для legal-evaluator test'ов
  - 100 заказов для OKK score calculation
- **Хранение:** `scripts/*_real_cases.fixture.json`

---

## Д

### Dead-letter queue (Мертвая очередь)
- **Определение:** Место, где оказываются job'ы, которые не прошли после N попыток
- **Примеры:** Transcription задачи, что не смогли скачать запись после 4 retry
- **Таблица:** `system_jobs` с `status = 'dead_letter'`

### Due Diligence (ДД)
- **Определение:** Проверка контрагента перед сделкой
- **Агент:** Борис (интеграция с Dadata, ФНС, реестры)
- **Результат:** Светофор (green/yellow/red risk)

---

## Е

### Event-driven (Событийно-управляемый)
- **Определение:** Система, где действия триггерят события, события создают job'ы, worker'ы их обрабатывают
- **Примеры:**
  - `recording_ready` → job `call_transcription`
  - `order_changed` → job `order_score_refresh`
- **Противоположность:** Batch (cron каждые N минут)

## З

### Zombie-сделка (Зомби-сделка)
- **Определение:** Заказ, который долго стоит в одном статусе (зависший)
- **Детектор:** Агент Анна через `semantic_check` анализирует комментарии
- **Действие:** Alert менеджеру в Telegram

---

## И

### Idempotency key (Ключ идемпотентности)
- **Определение:** Уникальный ключ, который гарантирует, что repeat request'ы не создают дубли
- **Пример:** `idempotency_key: "call_transcription:{callId}"` — если запустить дважды, старт только один раз
- **Таблица:** В `system_jobs` и `call_transcription`

### Intent routing (Маршрутизация интента)
- **Определение:** Определение, какой агент обработает запрос пользователя
- **Примеры:**
  - "Как вернуть товар?" → Дарья (Legal helpdesk)
  - "Как добавить контакт?" → Елена (Product KB)
  - "Проверьте контрагента" → Борис (Due Diligence)

## К

### Canonical table (Каноническая таблица)
- **Определение:** Single source of truth для доменного объекта
- **Примеры:**
  - `raw_telphin_calls` → для звонков
  - `orders` → для заказов
  - `order_metrics` → для context (но не основной)

### Cron (Crон-задача)
- **Определение:** Периодически запускаемая задача (по расписанию в Vercel)
- **Примеры:** `/api/cron/transcribe` каждые 10 минут
- **Сейчас:** Deprecated в пользу event-driven + fallback poller

---

## Л

### Lag (Отставание)
- **Определение:** Временная задержка между событием в источнике и его отражением в ОКК
- **Примеры:**
  - RetailCRM order changed → появляется в ОКК через 1-2 минуты
  - Recording ready → транскрибация завершена через 3-7 минут
- **Мониторинг:** `/api/settings/system-status`

### Lead (Лид)
- **Определение:** Потенциальный клиент, собранный через widget'ом
- **Агент:** Елена (Lead Catcher)
- **Процесс:** Елена собирает контакты → Семён создаёт заказ в CRM

---

## М

### Middleware (Middleware)
- **Определение:** Функция, обрабатывающая request'ы перед попаданием в route handler'ы
- **Примеры:** Auth middleware, Tenant context middleware
- **Файл:** `middleware.ts` в Next.js

### Multi-tenant (Мультитенантный)
- **Определение:** Система, обслуживающая множество независимых клиентов с изолированными данными
- **Принцип:** Каждый клиент (tenant) видит только свои данные

---

## Н

### Normalization (Нормализация)
- **Определение:** Приведение данных к стандартному формату
- **Примеры:**
  - Номер телефона: "+7-999-123-45-67" → "+79991234567"
  - Статус заказа: "ожидание оплаты" → "waiting_payment"

---

## О

### ОКК (Order Quality Control)
- **Определение:** Консультант для контроля качества заказов
- **Агенты:** Анна, Максим, Игорь, Семён
- **Цель:** Дать менеджеру 360° вид на заказ и риски

### Order metrics (Метрики заказа)
- **Определение:** Набор сведений о заказе (контекст, история, анализ)
- **Таблица:** `order_metrics` → поле `full_order_context`
- **Используется:** При каждом пересчёте score

---

## П

### Pipeline (Пайплайн)
- **Определение:** Последовательность обработки данных
- **Примеры:**
  - Real-time pipeline: webhook → job → worker → score → aggregate
  - Transcription pipeline: recording_ready → download → transcribe → semantic_rules

### Prompt (Промпт)
- **Определение:** Инструкция для ИИ (GPT)
- **Примеры:**
  - Редлининг контракта: "Анализируй этот контракт на red flags"
  - Классификация письма: "Определи тип входящего письма"

---

## Р

### RAG (Retrieval-Augmented Generation)
- **Определение:** Поиск релевантной информации + генерация ответа с контекстом
- **Примеры:**
  - Дарья ищет в KB ответ на вопрос → генерирует ответ с source'ом
  - Елена ищет в `product_knowledge` характеристики товара → отвечает

### Redlining (Редлининг)
- **Определение:** Анализ и маркировка проблемных пунктов в контракте
- **Агент:** Лев
- **Результат:** Отчет с red/yellow флагами и предложениями

### RBAC (Role-Based Access Control)
- **Определение:** Система прав доступа на основе ролей пользователя
- **Примеры:** account_owner может удалить аккаунт, analyst может только читать
- **Таблица:** `roles_and_permissions`

### RLS (Row-Level Security)
- **Определение:** Политика Supabase, фильтрующая rows по `account_id`
- **Назначение:** Безопасность — невозможно вернуть чужие данные
- **Примеры:** `SELECT * FROM orders WHERE account_id = current_account_id()`

---

## С

### Score (Скор/Оценка)
- **Определение:** Общая оценка качества заказа (от 0 до 100 или процент)
- **Таблица:** `okk_order_scores`
- **Компоненты:** Чеклист (30%), семантика (30%), правила (20%), приоритет (20%)

### Semantic check (Семантическая проверка)
- **Определение:** AI-анализ смысла комментариев и истории заказа
- **Агент:** Анна
- **Примеры:** "Это ли zombie-deal?", "Есть ли в комментах признаки проблемы?"

### Sync (Синхронизация)
- **Определение:** Обновление данных из внешней системы
- **Примеры:**
  - `retailcrm-order-delta` — дельта-синк заказов
  - `retailcrm-history-delta` — синк истории/комментариев
  - `telphin-fallback` — страховка для пропущенных webhook'ов

---

## Т

### Tenant context (Контекст тенанта)
- **Определение:** Текущий `account_id` и `user_id` в контексте request'а
- **Где:** Middleware извлекает из JWT, передаёт в handler'ы
- **Использование:** Все query'ы автоматически фильтруются по account_id

### TTL (Time To Live)
- **Определение:** Время, в течение которого job считается "живым" (processing)
- **Примеры:** Если job в processing дольше 30 минут → watchdog вернёт в queued
- **Таблица:** `system_jobs.lock_expires_at`

---

## У

### Unread (Непрочитанное)
- **Определение:** Количество непрочитанных сообщений в мессенджере
- **Таблица:** `messenger_presence` и badge на UI

---

## Ф

### Feature flag (Флаг функции)
- **Определение:** Toggleable включение/отключение функции без redeploy
- **Примеры:** `use_ai_classifier`, `enable_telegram_sync`, `realtime_pipeline_enabled`
- **Таблица:** `feature_flags`

### Fallback (Fallback/Страховка)
- **Определение:** Резервный путь обработки при недоступности основного
- **Примеры:**
  - Webhook недоступен → fallback poller
  - OpenAI timeout → graceful degradation (score без insights)

## Ч

### Чеклист (Checklist)
- **Определение:** Набор жёстких проверок, которые выполняются для каждого заказа
- **Примеры:** "ТЗ получено?", "ИНН валиден?", "Фото товара загружено?"
- **Отличие от правил:** Чеклист встроен в код, правила настраиваются в UI

---

## Ш

### Widget (Виджет)
- **Определение:** Встраиваемый элемент UI на сайте клиента
- **Примеры:** OKK Chat Widget для консультирования
- **Особенность:** Изолированный Shadow DOM, не влияет на стили сайта

---

## Э

### Eskalation (Эскалация)
- **Определение:** Передача задачи более высокому уровню (ИИ → юрист, ИИ → manager)
- **Примеры:** Дарья не знает ответ → создаёт задачу юристу

- **Определение:** Pipeline обработки правовых задач (helpdesk, redlining, due diligence, claims)
- **Агенты:** Дарья, Лев, Борис, Григорий
