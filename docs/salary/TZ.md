# ТЗ: модуль расчёта зарплаты в сервисе ОКК

**Проект:** OKKRiteilCRM (контроль качества + RetailCRM)
**Что добавляем:** блок расчёта ЗП менеджеров отдела продаж
**Статус:** черновик для совместной разработки в Claude Code
**Период внедрения:** июль 2026 (первый расчётный месяц по новой мотивации)

---

## 0. Зачем

Сейчас зарплата считается вручную в Google-таблице «Мотивация». Логика построена вокруг маржи, но маржа всплывает только на отгрузке, а менеджеры почти не влияют на сумму заказа (клиенту нужно 10 шкафов — продают 10). Поэтому мотивацию переносим в плоскость **количества и качества обработанных заявок**, а маржа становится малой долей.

ОКК уже синхронизирован с RetailCRM и уже считает качество обработки — это половина нужных данных. Задача — добавить поверх этого расчётный движок ЗП и заменить им ручную таблицу.

---

## 1. Граница модуля

**Входит в задачу:**
- Сбор расчётных метрик из уже синхронизированных данных CRM (заявки, закрытия, выручка) и из скоринга ОКК.
- Конфигурируемые ставки/пороги/оклады с версионированием по дате.
- Движок расчёта ЗП по формуле ниже.
- UI: дашборд по периоду, drill-down по менеджеру, настройки, экспорт в Excel.
- Блокировка периода после выплаты + лог изменений.

**Не входит (отдельные задачи):**
- Мотивация расчётчиков (другая модель — скорость/точность расчётов + доля выигранных заявок).
- Бухгалтерская выгрузка/начисления — только расчёт «к выплате», без интеграции с ЗП-учётом.

---

## 2. Формула ЗП (ядро)

```
ЗП = Оклад
   + [ (Премия_за_заявки × К_качества) + Конв_бонус + Маржа_бонус ] × К_команды
   + Дежурства
```

Порядок важен:
1. К_качества (0,8–1,2) множит **только** премию за заявки.
2. К_команды множит **всю переменную часть** (заявки + конверсия + маржа).
3. Оклад и дежурства командным коэффициентом **не режутся**.

---

## 3. Компоненты и источники данных

| # | Компонент | Логика | Источник |
|---|---|---|---|
| 1 | **Оклад** | Фикс на месяц, пропорционально отработанным дням | Конфиг (на менеджера) + табель |
| 2 | **Премия за заявки** | Σ закрытых заявок × ставка по типу | RetailCRM: заказы со статусом «оплачен» |
| 3 | **К_качества** | Скоринг ОКК → коэф 0,8–1,2 | Модуль ОКК (уже считает) |
| 4 | **Конв-бонус** | Тир по конверсии новых заявок | RetailCRM: новые входящие заявки + закрытия |
| 5 | **Маржа-бонус** | Фикс 5 000 ₽, если личная маржа отгрузок ≥ ориентира | Себестоимость на этапе **отгрузки** (источник уточнить) |
| 6 | **Дежурства** | Кол-во смен × ставка | Табель дежурств (ручной ввод или модуль ОКК) |
| 7 | **К_команды** | Тир по выручке всего отдела за месяц | RetailCRM: сумма закрытий всех менеджеров без НДС |

---

## 4. Параметры (всё в конфиг, не хардкод)

Стартовые значения на июль 2026. Меняются в настройках, **версионируются по дате** (см. §8).

**Оклад:** 35 000 ₽/мес (база, на менеджера).

**Ставки за закрытую заявку (п.2):**
| Тип закрытой заявки | Ставка |
|---|---|
| Новый клиент | 2 000 ₽ |
| Постоянный клиент | 1 000 ₽ |
| Печь / ВТО (длинный цикл) | 3 000 ₽ |

**К_качества (п.3)** — взвешенный скоринг 0–100% → коэффициент:
| Скоринг | Коэф |
|---|---|
| ≥ 90% | 1,2 |
| 75–89% | 1,1 |
| 60–74% | 1,0 |
| 40–59% | 0,9 |
| < 40% | 0,8 |

Состав скоринга (веса уточнить по тому, что уже меряет ОКК):
| Метрика | Вес |
|---|---|
| Скорость первого ответа на заявку | 25% |
| Соблюдение регламента (КП вовремя, повторный контакт) | 25% |
| Скоринг качества звонков | 20% |
| Полнота заполнения CRM | 20% |
| Отсутствие «нулёвых» (необработанных) заявок | 10% |

Жёсткое правило: любая брошенная/необработанная заявка обнуляет последний подпункт.

**Конв-бонус (п.4)** — конверсия = закрытые новые / новые входящие за период:
| Конверсия новых | Бонус |
|---|---|
| < 25% | 0 |
| 25–34% | 3 000 ₽ |
| 35–44% | 6 000 ₽ |
| ≥ 45% | 9 000 ₽ |

Защита от малого знаменателя: бонус начисляется только при **≥ 10 новых заявок** за период (иначе 0). Порог в конфиг.

**Маржа-бонус (п.5):** 5 000 ₽, если личная маржа по **отгруженным** за месяц ≥ ориентира (старт: 1 500 000 ₽, в конфиг; обсуждаемо — фикс или % плана).

**Дежурство (п.6):** 250 ₽/смена.

**К_команды (п.7)** — выручка отдела без НДС за месяц:
| Отдел за месяц | Коэф |
|---|---|
| < 12 млн | 0,5 |
| 12–15,9 млн | 1,0 |
| 16–19,9 млн | 1,15 |
| ≥ 20 млн | 1,3 |

## 5. Классификации и определения (зафиксировать однозначно)

- **«Закрытая заявка»** = заказ в статусе оплаты (какой именно статус/статусы в RetailCRM считать закрытием — уточнить по справочнику статусов).
- **«Новый» vs «постоянный» клиент** — по истории заказов клиента в CRM: постоянный = ≥ 2 оплаченных заказа за последние N месяцев (N в конфиг). Уточнить, есть ли в CRM готовый флаг.
- **«Печь / ВТО»** — по категории товара/полю заказа. Нужен маппинг категорий каталога → этот тип.
- **«Новая входящая заявка»** — лид/заказ, созданный в периоде, канал = входящий с сайта (исключить авито, колл-центр, базу тендеров — в текущей таблице они помечены «не считаем»). Нужен фильтр по источнику.
- **Приведение к «без НДС»** — по налоговой категории сделки (в таблице видно: 5% → делим на 1,05; 0% для ПОБТ/ЗВТО — без изменений). Зафиксировать правило нормализации для командной выручки.

## 6. Краевые случаи

- **Отгрузка позже месяца оплаты.** Заявочная часть (п.2) начисляется по **дате оплаты**; маржа-бонус (п.5) считается по **дате отгрузки** (т.е. может относиться к сделкам прошлых месяцев). Две разные «точки отсечки» в одном расчёте.
- **Нет себестоимости на момент расчёта.** Отгрузка без заполненной себестоимости в маржу-бонус не попадает; при появлении данных — пересчёт (если период ещё открыт) или корректировка в следующем периоде.
- **Отмена / возврат после оплаты.** Корректировка применяется к периоду, в котором произошла отмена: если период открыт — пересчёт; если закрыт — отрицательная корректировка в следующем периоде. Логируется.
- **Сотрудник нанят/уволен в середине месяца.** Оклад — пропорционально дням; пороги конверсии — без изменений (но порог «≥10 заявок» защищает от неполного месяца).
- **Дубли заявок** (в таблице активно помечаются «дубль NN»). Дубли не считаются ни в заявки, ни в конверсию — нужен признак дубля в CRM или правило дедупликации.

## 7. UI

**Экран «Зарплата»:**
- Выбор периода (месяц). Статус периода: открыт / закрыт.
- Таблица по менеджерам: колонки = компоненты (оклад, премия за заявки, К_качества, конв-бонус, маржа-бонус, дежурства, К_команды, **итого к выплате**).
- Строка-итог по отделу + плашка «выручка отдела / К_команды / ФОТ как % от маржи».
- Кнопки: «Пересчитать», «Закрыть период», «Экспорт в Excel».

**Drill-down по менеджеру:**
- Список засчитанных заявок (новые/постоянные/печь) с суммами и датами.
- Скоринг качества с разбивкой по подпунктам.
- Конверсия: новые заявки / закрытые / %.
- Маржа отгрузок месяца.
- Дежурства.

**Настройки (только руководитель):**
- Редактирование всех параметров из §4.
- История версий конфига (кто, когда, что изменил).

**Контрольная сверка:** экспорт должен повторять структуру текущей гугл-таблицы, чтобы сверить расчёт за май/июнь до запуска.

---

## 8. Версионирование, блокировка, аудит

- **Конфиг effective-dated.** У каждой ставки/порога есть дата начала действия. Прошлые периоды считаются по конфигу, действовавшему в том месяце — изменение мотивации не ломает историю.
- **Блокировка периода.** После «Закрыть период» расчёт неизменяем; все правки — только через корректировки в следующем периоде.
- **Лог.** Все пересчёты, корректировки и изменения конфига пишутся в аудит-лог.

---

## 9. Права доступа

- Владелец / руководитель ОП: всё (расчёт, настройки, закрытие периода).
- Менеджер: read-only только своя ЗП и своя разбивка.
