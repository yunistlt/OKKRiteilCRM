## Зачем

Сейчас блок «Премия за заявки» (`premia_zayavki`) смешивает две независимые оси
в одном enum из трёх взаимоисключающих значений:

| Ось | Источник | Значения |
|-----|----------|----------|
| Тип клиента | история сделок клиента | новый / постоянный |
| Категория товара | кастомное поле заказа `typ_castomer` | печь/ВТО |

Печь/ВТО «перебивает» тип клиента (`classifyOrderType` в `lib/salary/metrics.ts`
делает `return 'pech_vto'` до проверки истории клиента). Категория товара —
отдельная сущность и должна жить в отдельном настраиваемом блоке.

## Решения (согласовано)

1. **`premia_zayavki`** остаётся только с типами клиента: `new` / `permanent`.
2. **Новый блок** — премия за категории товаров, настраивается админом.
3. Категория определяется **по заказу** (поле `typ_castomer`), не по позициям —
   у позиций (`raw_payload.items[]`) нет категории/группы, только `offer.name`.
   Покомпонентный расчёт = отдельная большая доработка (синк каталога товаров),
   вне этого объёма.
4. Список категорий для выбора админом — **синк словаря `typ_castomer` из
   RetailCRM** в таблицу `retailcrm_dictionaries`.
5. Печь/ВТО становится просто первой строкой нового блока (режим «Сумма», 3000 ₽)
   — итоговые суммы выплат не меняются, т.к. корзины и так непересекающиеся.
6. У каждой категории — **переключатель режима**:
   - **Сумма**: `кол-во заявок категории × ставка ₽`
   - **% от продажи**: `Σ выручка заявки × %`
   - **Коэффициент**: множитель на **всю переменную часть** зарплаты

## Архитектурное следствие: ДВА вычислительных блока

`compose()` маршрутизирует блок либо как аддитивный (читает `amount`), либо как
множитель (читает `multiplier`) — по `kind`. Один блок не может одновременно
давать и сумму, и множитель. Поэтому режимы раскладываются на два блока движка,
которые в UI можно показать как один раздел «Категории товаров»:

### Блок A — `premia_categories` (аддитивный)
- `kind: 'premia'`, `group: 'premia'` → умножается на К_качества и К_команды,
  как сейчас печь/ВТО (сохраняет текущие суммы).
- Обрабатывает строки с режимом **Сумма** и **% от продажи**.
- `params: { rows: [{ category: string, mode: 'sum' | 'pct', value: number }] }`
- compute:
  - `sum`:  `amount += countByCategory[cat] × value`
  - `pct`:  `amount += revenueByCategory[cat] × value / 100`

### Блок B — `coef_categories` (множитель)
- `kind: 'multiplier'`, `multiplierScope: 'variableBracket'` → перемножается с
  К_команды на всю переменную скобку.
- Обрабатывает строки с режимом **Коэффициент**.
- `params: { rows: [{ category: string, coef: number }] }`
- compute: `multiplier = Π coef[cat]` по категориям, в которых у менеджера есть
  хотя бы одна засчитанная заявка (несколько категорий → перемножение,
  как уже устроены множители variableBracket). Нет таких заявок → ×1.

> Открытый момент для подтверждения: триггер режима «Коэффициент» — «есть ≥1
> заявка категории». Альтернативы: порог по количеству/доле. По умолчанию берём
> «≥1 заявка» и перемножение нескольких категорий.

## Изменения в метриках

`buildPeriodMetrics` (`lib/salary/metrics.ts`) сейчас агрегирует
`countsByType = {new, permanent, pech_vto}`. Добавить агрегаты по произвольной
категории заказа:
- `countsByCategory: Record<string, number>` (по `typ_castomer`)
- `revenueByCategory: Record<string, number>` (выручка без НДС по категории)

`classifyOrderType` перестаёт спец-обрабатывать печь/ВТО — остаётся только
`new` / `permanent` (история клиента). Категория берётся напрямую из
`row.typ_castomer` для новых агрегатов.

## Синк словаря категорий

- В существующий синк справочников RetailCRM (`retailcrm_dictionaries`,
  миграция `20260102_retailcrm_dictionaries.sql`) добавить выгрузку словаря
  кастом-поля `typ_castomer` (`entity_type='customField'`,
  `dictionary_code='typ_castomer'`).
- Админ-UI блока читает категории из этого словаря (`item_code` + `item_name`).

## Совместимость и миграция

- **Сохранённые схемы** со старым `premia_zayavki.rates.pech_vto`:
  `premia_zayavki.compute` игнорирует лишний ключ (читает только `new`/`permanent`);
  миграция переносит `pech_vto`-ставку в строку `premia_categories` (режим «Сумма»).
- **Дефолтная схема** (сид) обновляется: `premia_zayavki` без `pech_vto` +
  `premia_categories` со строкой печь/ВТО = 3000 ₽.
- **Legacy-колонка** `salary_calc.premia_zayavki`: остаётся = клиентская премия.
  Категорийная премия попадает в `breakdown.blockContributions` (детализация).
  При необходимости — отдельная колонка/строка на дашборде (доработка UI).
- Все миграции additive, с `effective_from` (политика модуля ЗП).
