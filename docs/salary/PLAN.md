# Зарплатный модуль — план реализации

Опирается на [TZ.md](TZ.md) и [DECISIONS.md](DECISIONS.md). Первый расчётный месяц — июль 2026.

Главный принцип (повтор): **ноль хардкода**. Все ставки/веса/пороги/тиры/определения — в БД, effective-dated, редактируются в админке (admin/rop). Стартовые значения §4 ТЗ заливаются сидом как **данные**.

---

## 1. Решение по схеме конфига (ранее отложено — фиксирую здесь)

**Выбрано: одна effective-dated таблица `salary_config` с `key` + `value JSONB`.** Причина: один источник версионирования и аудита; скаляр, таблица-ставка и тировая шкала ложатся в `value` единообразно; минимум миграций при появлении новых параметров. Типобезопасность обеспечиваем на уровне приложения через Zod-схемы на каждый ключ. Если возражений нет — двигаемся на нём.

## 2. Схема данных (миграция Фазы 1)

Префикс таблиц — `salary_`. Точный DDL — в `migrations/20260608_salary_schema.sql`.

| Таблица | Назначение | Ключевые поля |
|---|---|---|
| `salary_config` | effective-dated конфиг (все параметры мотивации + определения) | `key`, `value jsonb`, `effective_from date`, `created_by`, `created_at`, `note` |
| `salary_period` | расчётный месяц и его статус | `year`, `month`, `status` (`open`/`closed`), `closed_at`, `closed_by` |
| `salary_calc` | результат расчёта на менеджера/период | `period_id`, `manager_id`, компоненты, `total`, `breakdown jsonb`, `computed_at`; `unique(period_id, manager_id)` |
| `salary_adjustment` | ручные корректировки (возвраты/отмены в закрытом периоде) | `period_id`, `manager_id`, `amount`, `reason`, `created_by` |
| `salary_duty` | дежурства и табель (ручной ввод в ОКК) | `manager_id`, `work_date`, `kind` (`duty`/`worked_day`), `shifts`, `created_by` |
| `salary_audit_log` | аудит: изменения конфига, пересчёты, закрытие периода, корректировки | `entity`, `action`, `actor`, `old_value jsonb`, `new_value jsonb`, `created_at` |

**Ключи `salary_config`** (значение в скобках = форма JSONB):
- `oklad` (скаляр), `rate_zayavka` (map `{new, permanent, pech_vto}`), `k_quality_tiers` (массив `{min, k}`), `conv_bonus_tiers` (массив `{min, bonus}`), `conv_min_zayavki` (скаляр), `discount_bonus` (`{metric, threshold, bonus}` — метрика выбирается, см. открытый вопрос), `duty_rate` (скаляр), `k_team_tiers` (массив `{min, k}`).
- Определения-справочники: `closing_status` (код статуса «Передано в производство»), `permanent_client_threshold` (скаляр, >2), `source_exclusions` (список orderMethod), `category_pech_vto_map` (список кодов категорий), `nds_normalization` (правила приведения к «без НДС»: 5%→/1.05, 0%→как есть).

Разрешение конфига «на период»: берём последнюю версию каждого ключа, где `effective_from <= начало периода`. История прошлых месяцев не ломается при смене мотивации.

## 3. Слой сбора метрик (`lib/salary/metrics.ts`)

Чистые функции «период → данные», из уже синхронизированных таблиц. Без записи в внешние системы.

- **Засчитанные заявки**: заказы, у которых в `order_history_log` есть переход в `closing_status` внутри периода. Тип каждой: «новый/постоянный» по числу оплаченных заказов клиента за всё время (>2 = постоянный), «печь/ВТО» по `category_pech_vto_map`. Ставка из `rate_zayavka`.
- **Конверсия**: закрытые новые / входящие за период (входящие = заказы, созданные в периоде, источник не из `source_exclusions`). Допуск при `≥ conv_min_zayavki`.
- **Скидочная дисциплина**: метрика из `discount_bonus.metric` по `raw_payload.items[].discount` / скидке заказа.
- **К_качества**: `AVG(okk_order_scores.total_score)` по менеджеру за период → коэф по `k_quality_tiers`.
- **Выручка отдела**: Σ засчитанных заказов всех менеджеров, приведённая к «без НДС» по `nds_normalization` → коэф по `k_team_tiers`.
- **Дежурства/табель**: из `salary_duty`.

## 5. Движок расчёта (`lib/salary/engine.ts`)

Чистая функция: `(период, конфиг_на_период, метрики) → salary_calc[]`. Реализует формулу:
```
ЗП = Оклад
   + [ (Премия_за_заявки × К_качества) + Конв_бонус + Бонус_за_скидочную_дисциплину ] × К_команды
   + Дежурства
```
Порядок: К_качества множит только премию за заявки; К_команды — всю переменную часть; оклад и дежурства не режутся. Оклад — пропорционально отработанным дням. Полный `breakdown` (засчитанные заказы, конверсия, скоринг, скидка) пишем в `salary_calc.breakdown` для drill-down и сверки.

Краевые случаи (§6 ТЗ): отмена/возврат → корректировка в `salary_adjustment` (открытый период — пересчёт; закрытый — в следующий); найм/увольнение в середине месяца → пропорция оклада; дубли не считаются.

## 6. API и UI

**API** (`app/api/salary/*`, защита через RBAC + фильтр менеджера на «свои»):
- `GET /api/salary?period=YYYY-MM` — таблица расчёта (admin/rop — все; manager — только своё).
- `POST /api/salary/recalc` — пересчёт открытого периода.
- `POST /api/salary/close` — закрытие периода.
- `GET /api/salary/export?period=` — экспорт в Excel (lib `xlsx`, структура повторяет гугл-таблицу для сверки).
- `GET/PUT /api/salary/config` — чтение/правка конфига (admin/rop), пишет новую версию + аудит.
- `GET/POST /api/salary/duty` — дежурства/табель.

**UI** (`app/salary/`):
- Экран «Зарплата»: выбор месяца, статус периода, таблица по менеджерам (компоненты + итого), строка-итог по отделу + плашка «выручка / К_команды / ФОТ как % маржи», кнопки «Пересчитать / Закрыть период / Экспорт».
- Drill-down по менеджеру: засчитанные заявки, скоринг с разбивкой, конверсия, скидочная дисциплина, дежурства.
- Настройки (admin/rop): редакторы скаляров, таблиц-ставок и **тиров** (добавить/удалить строку, двигать границы) + история версий конфига.
- Read-only «Моя зарплата» для роли manager.
- Модалка ввода дежурств в разделе ОКК.

**RBAC** ([lib/rbac.ts](../../lib/rbac.ts), `DEFAULT_ROUTE_RULES`):
- `/salary` + `/api/salary` → `['admin','rop']`.
- `/salary/my` + `/api/salary/my` → `['admin','rop','manager']`.

## 7. Версионирование, блокировка, аудит (§8)
- Конфиг effective-dated → прошлые периоды считаются по своей версии.
- «Закрыть период» → `salary_calc` неизменяем; правки только через `salary_adjustment`.
- Все пересчёты/корректировки/изменения конфига → `salary_audit_log`.
