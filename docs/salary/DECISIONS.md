# Зарплатный модуль — журнал решений

Дополняет [TZ.md](TZ.md). Здесь фиксируются принятые развилки и принципы. Код этому документу подчиняется.

## 🔴 Главный принцип: ноль хардкода

**Все ставки, веса, пороги, тиры, коэффициенты, оклады, лимиты — ТОЛЬКО в БД, редактируются в админке, версионируются по дате (effective-dated). В коде — ни одного зашитого числа из мотивации.**

Сюда входит всё:
- оклад, ставки за заявку (новый/постоянный), ставки премии за категории товара, ставка дежурства;
- вилка К_качества (0,8–1,2) и её пороги по скорингу;
- тиры и суммы конв-бонуса + минимум заявок для допуска;
- сумма и ориентир маржа-бонуса;
- тиры К_команды **включая нижнюю ставку (бывш. 0,5)**;
- веса подпунктов скоринга (если когда-нибудь перейдём на пересборку);
- определения: какой статус = «закрытие», порог «постоянного клиента» (≥N заказов за M месяцев), список категорий товара для премии (из справочника RetailCRM), список источников-исключений (авито/КЦ/база).

Архитектурное следствие: даже «определения» (статусы, маппинги, пороги) — это конфиг в БД, а не константы в `.ts`. Стартовые значения из §4 ТЗ заливаются сидом/миграцией как **данные**, а не как дефолты в коде. Числа в коде допустимы только как технический fallback на случай пустого конфига, и то под вопросом — лучше падать явной ошибкой «конфиг не задан».

## Технические находки из кода (раунд 2)

**Признаки «брошенной заявки» — что уже умеет ОКК (`lib/okk-evaluator.ts`):**
- (а) просрочка даты след. контакта — **есть**: метрика `next_contact_not_overdue`, [okk-evaluator.ts:800](../../lib/okk-evaluator.ts) (сравнивает `customFields.next_contact_date`/`data_kontakta` с сегодня).
- (в) в статусе дольше нормы — **есть, но норма захардкожена `< 5 дней`**: `deal_in_status_lt_5_days`, [okk-evaluator.ts:822](../../lib/okk-evaluator.ts). 🔴 Нарушает «ноль хардкода» — норму надо вынести в конфиг / тянуть из настроек RetailCRM.
- (б) необоснованный перенос даты — **только частично**: `reschedule_policy_check` в [rule-engine.ts:374](../../lib/rule-engine.ts) проверяет лишь **факт наличия** комментария при недозвоне (бинарно), **без семантической оценки осмысленности**. Для твоего определения нужен AI-анализ текста (инфраструктура для этого в ОКК есть — `analyzeText`).

**Закупочная цена / маржа:**
- Состав заказа `items[]` хранится **только целиком в `orders.raw_payload`** (JSONB), структурно не разложен — [retailcrm-orders.ts:56](../../lib/retailcrm-orders.ts).
- Закупочная цена позиций **в коде сейчас НЕ парсится** (хотя в UI RetailCRM она есть и в API v5 приходит как `items[].purchasePrice`). Цена позиции местами достаётся (`initialPrice`, `prices[].price`), закупка — нет.
- Расчёта маржи/прибыли в коде **нет вообще**. Для ФОТ надо парсить `raw_payload.items[].purchasePrice` (на лету или в новую `order_items`).

**Дата перехода в «Передано в производство»:**
- История статусов есть в `order_history_log` (`field='status'`) — [prioritization.ts:95](../../lib/prioritization.ts); плюс `statusUpdatedAt` в `raw_payload` и события в `raw_order_events`. Дату перехода в нужный статус достать можно.

### Риск маржа-бонуса при ручной закупке (на обсуждение)

Закупку вбивает менеджер, от маржи зависит его бонус → стимул занижать закупку. Маржа на ручном вводе — **подделываемый** менеджером в свою пользу показатель. **Скидка же — наблюдаемый и неподделываемый** (занизить скидку = реально дать меньше скидок, ровно то поведение, что мы хотим). Раз цель бонуса по словам заказчика — «продавать без скидок», предлагается привязать бонус к скидочной дисциплине (средняя скидка ≤ X% / доля сделок без скидки), а маржу считать для аналитики и плашки «ФОТ как % маржи», но не делать её триггером бонуса. **Решение не принято — ждёт обсуждения.**

### Доработки ОКК-скоринга (вытекают из решения #12)
- Вынести норму «дней в статусе» из хардкода `< 5` → нормы из RetailCRM по каждому статусу.
- Добавить AI-оценку осмысленности комментария при переносе даты след. контакта (использовать `analyzeText`).
- После доработки сверить, что `total_score` корректно «обнуляется/проседает» на брошенных заявках.

## Итоговая модель (актуальная, заменяет §2–3 ТЗ где расходится)

**Формула** (без изменений по структуре):
```
ЗП = Оклад
   + [ (Премия_за_заявки × К_качества) + Конв_бонус + Бонус_за_скидочную_дисциплину ] × К_команды
   + Дежурства
```

**Компоненты и источники (актуально):**
| Компонент | Логика | Источник | Статус данных |
|---|---|---|---|
| Оклад | фикс/мес, пропорц. отработанным дням | конфиг + табель (ручной ввод в ОКК) | табель — новый экран |
| Премия за заявки | Σ засчитанных заявок × ставка по типу | заказы, вошедшие в статус **«Передано в производство»** | ✅ есть (raw_payload + order_history_log) |
| К_качества | `AVG(total_score)` по менеджеру за месяц → коэф по тирам | `okk_order_scores.total_score` | ✅ есть; требует доработки 3 признаков «брошенной» |
| Конв-бонус | тир по конверсии (закрытые новые / входящие за месяц) | заказы + источник заявки | ✅ есть |
| Бонус за скидочную дисциплину | средневзвеш. %% скидки по заказам месяца ≤ threshold → bonus | `raw_payload.items[].discountTotal` / `initialPrice` | ✅ данные есть, парсинг достроить (Фаза 3) |
| Дежурства | смены × ставка | ручной ввод в ОКК (модальное окно) | новый экран + таблица |
| К_команды | тир по выручке отдела за месяц (без НДС) | сумма засчитанных заказов всех менеджеров | ✅ есть |

**Тип заявки** (для ставки премии): «новый» = 1–2-й оплаченный заказ клиента за всё время, «постоянный» = 3-й и далее (считаем по истории). Категория товара заказа (`customFields.typ_castomer`) — отдельно, для добавочной премии по категориям (блоки `premia_categorii`/`coef_categorii`). Все ставки/пороги — в конфиг.

**Структура скидок/маржи в `orders.raw_payload` (подтверждено на заказе 53076):**
- Позиция `items[]`: `initialPrice` (база до скидки), `prices[0].price` (после), `discountTotal` (скидка по строке), `purchasePrice` (закупочная — для маржи), `quantity`, `vatRate`.
- Уровень заказа: `discountManualAmount` / `discountManualPercent` («Разовая скидка», может быть null), `summ`/`totalSumm` (итог).
- **Формулы (для Фазы 3):**
  - Стоимость товаров до скидки = `Σ initialPrice × quantity`.
  - Сумма скидок по заказу = `Σ items[].discountTotal` (+ `discountManualAmount`, если задан на уровне заказа).
  - % скидки заказа = сумма скидок / стоимость товаров.
  - Маржа заказа (аналитика) = `Σ (prices[0].price − purchasePrice) × quantity`.
  - Проверка на 53076: товары 515 169, скидки 58 255.35 (=11.3%), итого 456 913.65 — сходится с UI RetailCRM.

## Открытые вопросы
- (нет блокеров)
- Тиры конверсии (25/35/45%) могут быть выше фактических (~20–27% в 2020) — свериться при контрольном пересчёте.
