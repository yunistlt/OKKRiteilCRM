# ШТАТНОЕ РАСПИСАНИЕ И РУКОВОДСТВО КОМАНДЫ ОКК

Этот документ является единственным источником истины по ролям, терминологии и правилам работы системы ОКК.

---

## 👥 1. Состав команды и роли

### 📊 АННА (Бизнес-аналитик)
**РОЛЬ:** Стратегический анализ и поиск смыслов.
- **Обязанности:** Глубокий разбор сделок, поиск ЛПР, детекция «Зомби-сделок».
- **Инструментарий:** Модуль `semantic_check` (AI Анализ). Она «читает» комментарии и звонки.
- **Связь:** Передает инсайты Максиму для принятия решений.

### 🤓 МАКСИМ (Аудитор)
**РОЛЬ:** Контроль качества и итоговая оценка.
- **Обязанности:** Сводит результаты Семёна и Игоря, считает итоговый %, выявляет нарушения регламентов.
- **Инструментарий:** Управляет «Движком правил» (Rule Engine).
- **Связь:** Пишет в Telegram чат «Отдел продаж ЗМК» через бота @OKKzmk.

### 👮‍♂️ ИГОРЬ (Диспетчер)
**РОЛЬ:** Контроль SLA и алерты.
- **Обязанности:** Мониторинг сроков, «Светофор приоритетов», оповещение о критических сбоях.
- **Инструментарий:** Чистая логика (без AI) — время в статусе, просроченные задачи.
- **Связь:** Пишет лично руководителю @zmktlt.

### 🎧 СЕМЁН (Архивариус)
**РОЛЬ:** Синхронизация данных.
- **Обязанности:** Сбор данных из RetailCRM 24/7 (заказы, история, звонки).
- **Связь:** Обеспечивает свежими данными Анну, Максима и Игоря. Мы не ждем смены статуса, Семён качает «любой чих».

## 📖 2. Терминология: Правила vs Чеклист

### **Чеклист контроля (Общая сводка)**
- **Где:** Колонки на главном дашборде (ТЗ получено, ИНН есть и т.д.).
- **Как работает:** Жестко прописан в коде (`okk-evaluator.ts`).
- **Цель:** Мониторинг «здоровья» заказа и формирование общего % рейтинга.
- **Красный крестик:** Портит % рейтинга, но не создает отдельное «Нарушение».

### **Правила (Динамический контроль)**
- **Где:** Страница «Настройка правил».
- **Как работает:** Настраивается через интерфейс (или SQL), выполняется движком Максима.
- **Цель:** Создание конкретных **Нарушений** (Alerts) и уведомление в Telegram.
- **Штраф:** Вычитает баллы за совершённое нарушение.

## 🛠 3. Работа с Правилами

### Конструктор логики
Каждое правило состоит из:
1. **Триггер (Когда?)**: Например, `status_change` (смена статуса).
2. **Условие (Что?)**: 
   - `field_empty` (Техническая проверка ячейки в CRM).
   - `semantic_check` (Инструкция для Анны: проверить смысл текста).
3. **Параметры (Итог)**: Штраф, критичность, Telegram-уведомление.

### Создание через SQL (для разработчика)
```sql
INSERT INTO okk_rules (code, name, description, entity_type, logic, points, severity, is_active, notify_telegram)
VALUES (
  'rule_code', 'Название', 'Описание', 'order',
  '{"trigger": {"block": "status_change", "params": {"target_status": "code"}}, "conditions": [{"block": "field_empty", "params": {"field_path": "INN"}}]}'::jsonb,
  10, 'medium', true, true
);
```

## 🚀 4. Технические инструкции (Vercel & Cron)
- Вся фоновая работа (синхронизация звонков, матчинг, расчет приоритетов) управляется **Vercel Cron**.
- Локальный запуск используется только для разработки и ручного «докачивания» данных через кнопки в интерфейсе.
