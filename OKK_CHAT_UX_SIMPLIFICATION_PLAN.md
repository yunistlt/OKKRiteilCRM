# План реализации упрощения UX чата ОКК

## 1. Контекст

Сейчас в правой панели консультанта ОКК одновременно присутствуют:

- выбор ветки диалога;
- кнопки выбора глубины ответа short/full.

По текущему продуктному решению выбор ветки диалога нужен и соответствует привычному паттерну GPT-чата: пользователь должен видеть список начатых обсуждений и возвращаться в нужный контекст.

Переключатель глубины ответа в интерфейсе не нужен. Он создаёт лишнее решение для пользователя, усложняет шапку чата и визуально выглядит как обязательная настройка перед вопросом. Для рабочего сценария ожидается полноценный ответ по умолчанию.

## 2. Цель

Сделать чат понятным и предсказуемым:

- оставить выбор истории диалогов;
- убрать из UI выбор short/full;
- всегда возвращать полноценный ответ по умолчанию;
- сохранить возможность продолжать старые диалоги по темам и заказам.

## 3. Продуктовое решение

### Что остаётся

- Список диалогов в шапке или рядом с шапкой чата.
- Создание новой темы.
- Переход между ранее созданными обсуждениями.
- Привязка ветки к контексту: общий раздел или заказ, если заказный контекст реально используется.

### Что убирается

- Переключатель short/full.
- Любые подписи в UI, заставляющие пользователя выбирать глубину ответа до отправки вопроса.

### Новое поведение по умолчанию

- Ответ всегда формируется в полном рабочем формате.
- Если пользователю нужен сжатый ответ, он формулирует это естественным языком в сообщении.
- В интерфейсе не должно быть ощущения, что без выбора режима ответ будет неполным или неправильным.

## 4. Область изменений

### Frontend

- Упростить шапку панели консультанта.
- Переименовать селект веток в более понятную сущность: Чаты, История диалогов или Темы.
- Визуально усилить, что это именно список обсуждений, а не фильтр или режим.
- Оставить кнопку создания новой темы рядом со списком чатов.

### Backend

- Принять полный ответ как стандартное поведение по умолчанию.
- Не требовать responseMode от интерфейса для штатной работы.
- Сохранить обратную совместимость со старыми сообщениями, где metadata.responseMode уже записан.

### Данные и история

- Не ломать существующие threadId и историю сообщений.
- Не удалять старые записи только из-за того, что режим ответа больше не выбирается в UI.

## 5. Этапы реализации

### Этап 1. UX-упрощение панели

- Удалить блок Режим ответа из шапки чата.
- Оставить только управление ветками обсуждений.
- Проверить, что шапка остаётся компактной на desktop и mobile.

### Этап 2. Нормализация терминологии

- Заменить техническое ощущение ветка/режим на пользовательский язык чат/тема/история.
- Привести подписи и aria-label к одной терминологии.

### Этап 3. Упрощение клиентской логики

- Убрать из клиентского интерфейса состояние выбора short/full.
- Перестать отправлять responseMode из UI как обязательный параметр.
- Проверить, что новый чат, продолжение старого чата и быстрая отправка вопроса работают без регрессий.

### Этап 4. Защита совместимости на API

- Оставить серверный fallback на full по умолчанию.
- Не зависеть от наличия responseMode в запросе.
- Не ломать отображение старых сообщений, если у них есть metadata.responseMode.

### Этап 5. Финальная полировка

- Проверить пустое состояние чата.
- Проверить мобильную панель.
- Проверить визуальную иерархию шапки после удаления лишнего блока.

## 6. Критерии приёмки

- Пользователь видит список чатов и понимает, что может вернуться к предыдущему обсуждению.
- В интерфейсе больше нет выбора short/full.
- Любой новый вопрос получает полноценный ответ без дополнительных настроек.
- История существующих диалогов остаётся доступной.
- Переключение между чатами не ломает сообщения и контекст.
- На mobile и desktop шапка не выглядит перегруженной.

## 7. Риски

- Если где-то в продуктовой логике short/full использовался как скрытый технический переключатель, его удаление из UI нужно сопровождать проверкой API fallback.
- Если ветки по заказам сейчас не несут реальной пользовательской пользы, один только селект чатов без нормальной терминологии останется неочевидным.
- Если список чатов будет называться технически, пользователь продолжит путать его с фильтром или выбором заказа.

## 8. Рекомендуемое итоговое состояние

В панели консультанта остаются:

- имя агента и статус;
- список чатов или тем;
- кнопка новой темы;
- поле ввода сообщения.

Из панели убираются все настройки, которые не являются обязательными для старта разговора.
